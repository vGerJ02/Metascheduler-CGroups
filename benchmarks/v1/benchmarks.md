
## Hadoop Benchmarking Guide

This document explains the purpose, configuration, and interpretation of the common Hadoop benchmarks included in the `run_benchmarks.sh` script.

---

### **1. Overview**

Benchmarking a Hadoop cluster helps evaluate its **I/O performance**, **compute throughput**, and **MapReduce efficiency** under different workloads.

The benchmarks used here come from the official Hadoop example suite and test various components:

| Benchmark                             | Focus Area                             | Component Tested                               |
| ------------------------------------- | -------------------------------------- | ---------------------------------------------- |
| **TeraGen / TeraSort / TeraValidate** | End-to-end MapReduce performance       | MapReduce + HDFS                               |
| **TestDFSIO**                         | Disk I/O (read/write throughput)       | HDFS I/O layer                                 |
| **NNBench**                           | Metadata operations (file create/read) | NameNode performance                           |
| **MRBench**                           | Small repeated MapReduce jobs          | JobTracker/ResourceManager and task scheduling |
| **Sort**                              | Simple sorting workload                | Basic MapReduce functionality                  |

---

### **2. Benchmarks**

#### **2.1 TeraSort Suite**

This is a three-step benchmark that stresses both the **MapReduce framework** and **HDFS**.

1. **TeraGen** – Generates random data files of fixed-size records.

   ```bash
   hadoop jar hadoop-mapreduce-examples-*.jar teragen <num_records> <output_dir>
   ```

   * **`<num_records>`**: Number of 100-byte records to generate.

     * Example: `1000000` → roughly 100 MB of data.
   * **`<output_dir>`**: HDFS directory where data will be written.

   **Expected Output:** Multiple part files under the specified directory.

2. **TeraSort** – Sorts the data generated by TeraGen.

   ```bash
   hadoop jar hadoop-mapreduce-examples-*.jar terasort <input_dir> <output_dir>
   ```

   * Input: TeraGen output path
   * Output: Sorted dataset in another HDFS directory

   **Expected Output:** Sorted key-value pairs and job statistics including map/reduce duration.

3. **TeraValidate** – Checks whether the sorted output is correctly ordered.

   ```bash
   hadoop jar hadoop-mapreduce-examples-*.jar teravalidate <input_dir> <output_dir>
   ```

   **Expected Output:** “No errors detected” message if data is valid.

#### **2.2 TestDFSIO**

Measures **read/write throughput** and **I/O consistency** of the HDFS subsystem.

```bash
hadoop jar hadoop-mapreduce-client-jobclient-*-tests.jar TestDFSIO \
  -write -nrFiles <num_files> -fileSize <size> -resFile <result_file>
```

Parameters:

* **`-write`** or **`-read`**: Specifies the operation type.
* **`-nrFiles`**: Number of files to read/write (e.g., 10).
* **`-fileSize`**: File size per file (e.g., 100MB, 1GB).
* **`-resFile`**: Output file path for results.

**Expected Output:**
A summary section at the end showing:

```
--- TESTDFSIO TEST: write ---
Throughput mb/sec: 115.43
Average IO rate mb/sec: 112.87
```

These values indicate aggregate and average I/O performance.

---

#### **2.3 NNBench**

Evaluates the **NameNode’s metadata operation throughput** (file creation, reading, and deletion).

```bash
hadoop jar hadoop-mapreduce-client-jobclient-*-tests.jar nnbench \
  -operation <op> -maps <num_maps> -reduces <num_reduces> \
  -bytesToWrite <bytes> -numberOfFiles <num_files> -baseDir <dir>
```

Parameters:

* **`-operation`**: `create_write`, `open_read`, or `delete`.
* **`-maps` / `-reduces`**: Number of mapper/reducer tasks.
* **`-bytesToWrite`**: File size for each created file.
* **`-numberOfFiles`**: Number of test files to create.
* **`-baseDir`**: Base HDFS directory for test data.

**Expected Output:**
Summary of operation time and throughput, for example:

```
Total operations: 100
Avg time per operation (ms): 12
Throughput (ops/sec): 8.3
```

---

#### **2.4 MRBench**

Runs a series of small MapReduce jobs in succession to test **job submission latency** and **scheduler performance**.

```bash
hadoop jar hadoop-mapreduce-client-jobclient-*-tests.jar mrbench \
  -numRuns <n> -maps <m> -reduces <r>
```

Parameters:

* **`-numRuns`**: Number of test jobs (e.g., 5–10).
* **`-maps` / `-reduces`**: Number of mapper/reducer tasks per job.

**Expected Output:**
Job-level execution times for each run, showing consistency or variation in scheduling performance.

---

#### **2.5 Sort**

A basic MapReduce sorting benchmark for smaller clusters or smoke testing.

```bash
hadoop jar hadoop-mapreduce-examples-*.jar sort <input_dir> <output_dir>
```

**Expected Output:**
Sorted data in the output directory, along with standard job execution metrics.

---

### **3. Tuning Parameters**

When running benchmarks on different clusters, adjust the parameters for meaningful results:

| Parameter         | Effect                        | Recommendation                                               |
| ----------------- | ----------------------------- | ------------------------------------------------------------ |
| `TERAGEN_RECORDS` | Total dataset size            | Scale based on cluster size (e.g., 10 million for 3–5 nodes) |
| `DFSIO_FILE_SIZE` | File size per I/O test        | Larger sizes (e.g., 1GB) for testing real-world throughput   |
| `DFSIO_FILES`     | Number of test files          | Spread load across nodes (usually ≥ number of data nodes)    |
| `NNBENCH_FILES`   | Number of metadata operations | 100–1000 for realistic metadata stress                       |
| `MRBENCH_RUNS`    | Number of MapReduce test runs | At least 5 for averaging results                             |

---

### **4. Interpreting Results**

After running all benchmarks:

* **High throughput (MB/s)** and **low average job time** indicate good performance.
* **Large variance** between runs suggests scheduling or I/O contention.
* **NNBench** is sensitive to metadata latency — slow results may indicate NameNode saturation.
* **TeraValidate** errors imply configuration or HDFS consistency issues.

| Local log files        | `./benchmark_logs/`           |
| DFSIO result summaries | `/benchmarks/TestDFSIO_*.log` |

---

### **6. Best Practices**

* Always run benchmarks on an **idle cluster**.
* Clean up old benchmark data:

  ```bash
  hdfs dfs -rm -r /benchmarks/*
  ```
* Repeat each test at least **3 times** for consistent averages.
* Adjust JVM heap and YARN memory settings if tasks fail due to OOM errors.
